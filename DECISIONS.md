### 2026-07-04
**Decision:** Fixed a false-positive bug in `analyzeMAOutput`'s bold-detection regex. Any grammar pattern using a fill-in-the-blank marker like `___` (three or more underscores) was incorrectly flagged as bold formatting, because the naive `__[^_\n]+__` pattern matched the middle two underscores of a longer run. Added the same lookaround guards the italic check already used, so the pattern now only matches a genuine isolated `__word__` or `**word**` pair.
**Reason:** Diagnosed with certainty from a single real report (student "Jerry," Grammar Points section using `"___ needs a ___"`) — root cause was mechanical and confirmed, not something requiring the three-occurrence threshold. Classified as a Bug in the validator itself, not a protocol or model output issue. Verified against 5 test cases (the false-positive pattern, genuine `__bold__`, genuine `**bold**`, the full real report text, and a `____` edge case) with no regressions.
**Constitution article:** Bug Rule (fix confirmed bugs on sight, regardless of occurrence count — the Three Occurrence Rule applies to *whether a pattern is real*, not to bugs with a diagnosed, certain root cause); Article Zero (a validator producing false positives erodes trust in the "reviewed" signal, directly undermining the product's core value).

### 2026-07-04
**Decision:** Fixed a second false-positive bug in `analyzeMAOutput`'s star-count check, found the same day as the bold-detection fix. The check counted approved star emoji across the *entire* report instead of just the "Total Stars Today" section. Since the protocol allows any approved star emoji (⭐ 🌟 ✨ 💫 🌠) to also serve as a decorative vocabulary icon elsewhere (e.g. "Soul ✨ – ..."), a perfectly correct report with exactly 10 real stars was flagged as having 11. Scoped counting to only the text between the "Total Stars Today" heading and the next blank line; added an explicit warning for the case where that section can't be located at all, rather than silently miscounting.
**Reason:** Diagnosed with certainty from a real report (student "James") after the teacher manually confirmed exactly 10 stars were present. Classified as a Bug in the validator, not a protocol or model output issue — same category as the earlier bold-detection fix found the same day. Verified against 4 cases: the real false-positive report, a genuine 11-star over-count within the section, a genuine 9-star under-count, and a report missing the section entirely.
**Constitution article:** Bug Rule (confirmed bug, fixed on sight); Article Zero (false positives from validation directly undermine the "trusted judgment" product promise — two such bugs found in one day is itself a signal the validator needs closer scrutiny, not just individual patches).

### 2026-07-05
**Decision:** Replaced OF 1.0 (14-section full trial evaluation) with OF 1.1 (3-section complementary narrative: Strengths, Areas for Improvement, Learning Suggestions), before any real field-test lessons had been logged under OF 1.0. Removed Student Information, Lesson Focus, Overall Impression, the six per-skill assessment sections, Recommended Level, and Final Recommendation. Target length changed from ~1 page (soft upper-bound-only check) to 180-300 words (two-sided check). Validator rewritten to check for exactly 3 sections in order rather than 14 present-in-any-order.
**Reason:** The platform's own interface already captures rating, level recommendation, and other evaluation parameters manually. A 14-section AI-generated evaluation duplicated that work instead of complementing it — OF's actual job is narrative texture the platform's structured fields can't express, not a second full evaluation system. Since this predates any real OF field data, it was treated as a design correction rather than a field-test-driven change, but still versioned rather than silently edited, per the same discipline MA already follows.
**Constitution article:** Judgment evolves through evidence, but also through honest re-examination of the design before evidence accumulates on a wrong foundation — versioning makes that correction traceable rather than silent. Article Zero (a shorter, non-duplicative report that complements manual platform fields serves "transform lesson evidence into a high-quality report" better than a longer one that restates work already done elsewhere).

### 2026-07-05
**Decision:** Replaced OF 1.1's numeric 10-point tone table (shared in spirit with MA's Level Integration) with a qualitative 5-tier guidance framework (Low, Low/Medium, Medium, Medium/High, High) — OF 1.2. The shared rating selector in the UI is unchanged; the numeric 1-6 pick a teacher makes is still the only input, but for OF it's now translated internally into one of 5 tiers (two adjacent numeric levels per tier) before reaching the prompt, rather than driving tone directly via a 10-point table. Added a validator check for the two distinctive compound tier labels ("Low/Medium", "Medium/High") leaking into generated output — deliberately did not check for the plain words "Low"/"Medium"/"High" alone, since those appear constantly in ordinary English prose and would produce false positives rather than real signal.
**Reason:** Requested directly, with an explicit constraint that the tier must never surface in the output as a score, label, heading, or recommendation. That constraint is now enforced in three places: the static protocol text, the per-request runtime parameters block, and a mechanical (non-semantic) validator check — consistent with the earlier decision to defer semantic validation until real field data justifies it. This is pattern-matching on a small, distinctive string set, the same category of check as the sign-off or markdown checks, not the kind of open-ended semantic judgment (e.g. "does this discuss trajectory") that was explicitly deferred.
**Constitution article:** Facts vs. Judgment (the qualitative tier is Judgment-layer guidance computed from a Fact — the numeric rating — via a fixed mapping, inserted before the model ever sees the prompt, same architecture as MA's runtime parameters); Bug Rule / validation philosophy (mechanical checks only, semantic checks deferred until evidence).

### 2026-07-04
**Decision:** Defined an explicit "approved star collection" (⭐ 🌟 primary, ✨ 💫 🌠 for occasional variety) as a shared constant in `KidbusterCore`, used by both the prompt and the validator. Previously the protocol text referenced "the approved star collection" without ever defining one.
**Reason:** This closes a pre-existing spec gap rather than introducing new judgment — the protocol already presumed such a list existed. Classified as a Fact (a fixed list), not a Judgment change, so it did not require the three-occurrence evidence threshold. Implemented via the runtime parameters block, not by editing the immutable protocol text.
**Constitution article:** Facts vs. Judgment (deterministic list, inserted before prompt construction); Article Zero (removes ambiguity without adding complexity or changing report quality's underlying logic).

### 2026-07-04

**Reason:** The Constitution is ratified. The next source of truth is real classroom evidence, not further design discussion.
**Constitution article:** Article Zero (workload/quality objective); Layer 3 — Versioning (evidence-driven change only).

### 2026-07-04
**Decision:** `cleanTranscript()` added to `KidbusterCore` as the final engineering task before field testing began. Timestamp-stripping logic was corrected before shipping after testing revealed it would have removed legitimate content (e.g. "the time is 3:45") — fixed to only strip transcript-marker timestamps (whole-line or bracketed-prefix), never bare times inside natural sentences.
**Reason:** Required to clean real Krisp transcripts before prompt construction without violating "preserve real lesson content, do not over-clean."
**Constitution article:** Facts vs. Judgment (transcript cleaning is mechanical fact-preparation, not judgment); Article Zero (must not reduce report quality by discarding real content).

### 2026-07-05
**Decision:** Gave OF its own dedicated 5-tier rating control (Low, Low/Medium, Medium, Medium/High, High) in the UI, replacing the shared MA 1-6 numeric scale for OF. The teacher now picks the qualitative tier directly for OF reports instead of picking a MA-style number that gets silently translated into a tier behind the scenes. Removed the now-unused `OF_QUALITATIVE_TIER_FOR_LEVEL` lookup table entirely — `buildOFSystemPrompt` takes the tier as direct input. Added `OF_TIERS`, `ofTierColor`, `ofTierFillFraction` as OF's own equivalents of MA's `LEVELS`/`ratingColor`/`ratingFillFraction`. Draft persistence updated to save/restore which protocol was selected, not just which rating — needed once the two protocols stopped sharing one rating domain, since a saved OF tier value is meaningless against MA's button set and vice versa.
**Reason:** The previous design (shared numeric input, translated to a tier internally) was visually confusing: switching to OF still showed MA's 1-6 buttons, so it looked like OF hadn't actually changed anything even though the underlying prompt was correct. Making the tier the actual UI control removes an indirection that had no benefit once OF stopped exposing numeric ratings anywhere — the teacher was always going to think in terms of the qualitative tier for a trial-lesson narrative, so the control now matches the mental model directly.
**Constitution article:** Facts vs. Judgment (the tier is still a Fact computed before the prompt is built — it's just now supplied directly by the teacher rather than derived from a different Fact); Article Zero (a control that visibly reflects what's actually being sent serves "trusted judgment" better than one that quietly translates behind an unchanged-looking UI).

### 2026-07-05
**Decision:** Added a hard 4700-character ceiling to MA (version bump MA 2.1 → MA 2.2), plus a matching validator check in `analyzeMAOutput`. This is the first change driven by an actual field-test lesson rather than design reasoning alone.
**Reason:** A real James report (rating 5/5) came back at 5,232 characters, exceeding the delivery platform's hard 5000-character-per-comment limit by 232 characters. MA had no length constraint of any kind prior to this — unlike OF, which has had a 180-300 word target since 1.1. This was treated as immediate evidence, not a pattern requiring the Three Occurrence Rule: the platform's character limit is an external, deterministic Fact, not a judgment call about tone or quality, so a single confirmed violation of a hard technical constraint is sufficient grounds to act, the same way a diagnosed-with-certainty bug doesn't wait for a third occurrence. Verified against the actual over-limit report (correctly flagged) and a trimmed compliant version (correctly silent) before shipping.
**Constitution article:** Facts vs. Judgment (a platform character limit is a Fact about the world MA writes into, not a stylistic preference); Protocol Evolution Rule (shipped as a new named version, MA 2.2, not a silent edit to 2.1); Bug Rule / evidence threshold (a confirmed violation of a hard external constraint doesn't require repeated occurrences the way an ambiguous quality judgment would).

### 2026-07-05 (same day, follow-up)
**Decision:** Collapsed MA's two-tier length rule (4700 working target / 5000 absolute ceiling, from earlier today) into one single hard limit: 4800 characters, immutable, no exceptions. MA 2.3 → MA 2.4.
**Reason:** The two-tier structure was reasonable on its own terms but added complexity the situation didn't need — a report simply should never be so long it's flirting with the platform's true 5000 cap in the first place. One firm number below the real limit is simpler to reason about, simpler for the model to hit reliably, and removes the awkward case of a report technically passing a "working target" while still being uncomfortably close to the platform's actual ceiling.
**Constitution article:** Article Zero (simplicity that still fully satisfies the real constraint beats a more elaborate structure that hedges against the same constraint two different ways); Protocol Evolution Rule (versioned as 2.4, not a silent edit to 2.3).

### 2026-07-05 (same day, second follow-up)
**Decision:** Restored the two-tier length structure (soft working target + hard immutable ceiling), scaled proportionally to a lower hard limit: 4500 working target / 4800 absolute ceiling (previously 4700/5000). MA 2.4 → MA 2.5. Margin between the two tiers kept identical (300 characters) to the original design.
**Reason:** Preference for the two-tier warning granularity over the single-limit version shipped earlier today — a report at 4600 and a report at 4900 are different situations (one needs tightening, the other cannot be sent at all), and collapsing them into one undifferentiated "over the limit" message lost that distinction. Reintroduced with the lower ceiling requested, keeping the same absolute margin between target and ceiling that the original 4700/5000 pair used.
**Constitution article:** Protocol Evolution Rule (versioned as 2.5, not a silent edit); judgment evolves through iteration — the single-limit version wasn't wrong, it was a reasonable simplification that got revised once its tradeoff (losing the target/ceiling distinction) was weighed against the two-tier version's clarity.

### 2026-07-05 (OF)
**Decision:** Added a per-section character limit to OF (version bump 1.2 → 1.3): Strengths and Areas for Improvement must each individually stay under 1300 characters. Learning Suggestions has no equivalent limit.
**Reason:** These two sections are pasted into two separate fields on the delivery platform, each independently capped at 1500 characters — a genuinely different constraint shape than MA's single combined-report limit, since it's per-field, not per-report. 1300 leaves the same kind of safety margin under the real 1500 platform cap that MA's 4800/5000 pair established. Implemented as a real per-section extraction in the validator (isolating each section's actual body text between its header and the next section's header) rather than a whole-report character count, since a whole-report check couldn't distinguish which field would actually overflow. Verified against 5 cases: both compliant, each section individually over while the other stays clean, both over simultaneously (two distinct warnings), and the exact 1300 boundary.
**Constitution article:** Facts vs. Judgment (a platform field limit is an external Fact, same category as MA's constraint, not a style preference); Protocol Evolution Rule (versioned as 1.3, not a silent edit).

### 2026-07-05 (OF, continued)
**Decision:** Rewrote OF's Register and Audience section to substantially strengthen humanization (version bump 1.3 → 1.4). Restored "Never corporate. Never cold." — present in the original OF spec but missing from the current text, likely dropped during an earlier rewrite (the 1.0→1.1 three-section simplification or the 1.1→1.2 qualitative-tier change). Added new, more concrete guidance beyond the restored line: an explicit anti-generic-phrasing rule ("if a sentence could be copy-pasted into a different student's report without anyone noticing, it is too generic"), a directive to write as a real teacher describing a real student rather than a system completing a template, and light local reinforcement in the Strengths and Areas for Improvement template instructions specifically, since those are the two sections most likely to drift into generic filler.
**Reason:** The report is read directly by parents and school staff, not just validated internally — a clinical or form-like voice undermines the actual product, especially since OF was already built to avoid re-stating platform fields, meaning its only real content is the human narrative. Deliberately did not add a new automated validator check for this: "does this sound human" is a qualitative judgment with real false-positive/negative risk if approximated by pattern matching, exactly the kind of speculative semantic check that should wait for real field-test evidence before being built, not be invented preemptively.
**Constitution article:** Article Zero (a parent-facing report that reads as a form undermines the actual point of writing one at all); Bug Rule / regression awareness (a previously-specified requirement silently disappearing across version rewrites is worth catching and restoring, not just adding something new on top); deliberate restraint against speculative validator engineering, consistent with prior decisions to defer semantic checks until real usage data justifies them.

This and the three following MA/Sugarcoat prompt changes below should be reconciled into a proper version bump (e.g. V2.9 → V2.10) as a follow-up, rather than left as an undocumented gap between the code's literal version label and what the prompt actually says.

### 2026-07-06 (same day, follow-up)
**Decision:** Added a mandatory "LENGTH-TIER CONTENT RULE (IMMUTABLE)" to the Short and Medium tiers, on top of the numeric target/ceiling above. Short must: omit Pronunciation Focus entirely, drop the per-word "| Pronunciation: ..." segment from every Key Vocabulary line, include at most 1 grammar point (tightened from an initial 2), and include only 2 Grammar & Sentence Practice examples (down from the template's usual 3). Medium must cap grammar points at 3 and keep vocabulary Notes to 6 words or fewer. Long has no additional reduction. Added matching validator checks for all of these (Pronunciation Focus presence, per-word pronunciation field presence, grammar-point count via `👉` markers — deliberately format-agnostic across MA's numbered style and Sugarcoat's unnumbered one — and example count), not just the character-count check from the decision above.
**Reason:** Direct field evidence, not design reasoning alone. A real Short-tier generation (student "Oscar") came back at 2,599 characters against a 2,400 ceiling — over by 199 — while still including a full Pronunciation Focus section the rule was already supposed to omit. Breaking that real report down section-by-section showed Key Vocabulary alone cost 1,031 of the 2,599 characters (40%), and that a realistic combination of cuts (Pronunciation Focus omission + fewer grammar points + fewer examples) would have brought it to ~1,910 characters — comfortably under budget — without ever touching the shared, unconditional Key Vocabulary section. The rule was deliberately *not* used to cut Key Vocabulary itself, since that's shared, unconditional content across all three tiers and cutting it would mean Short showing genuinely less vocabulary detail, not just less prose — a bigger call than was asked for here.
**Constitution article:** Bug Rule (a single confirmed real-world overage from an actual generation is sufficient grounds to act, the same category as the earlier 4700/5000 MA-length and per-word Pronunciation false-positive fixes — this wasn't a hypothetical, it was a report that actually happened); Facts vs. Judgment (the character/word counts driving each cut were measured from real report text, not estimated).

### 2026-07-06
**Decision:** Added Blitz Protocol V1.0 as a fourth official protocol — a short (70-120 word target, 150-word hard ceiling), neutral, professional platform comment, distinct in kind from MA/Sugarcoat's long structured reports and OF's trial narrative. Built around a 10-entry "Model Pool" of internal writing-style variants; exactly one is selected per generation via a shuffle bag (not plain `Math.random()`), guaranteeing all 10 are used once before any repeat rather than risking visible streaks. Includes its own validator (word count, no-emoji via Unicode property matching, no bullets/headings, no leftover template placeholders, and a phrase-shingle check against near-verbatim copying from the selected model's own example text) and its own yellow theme/logo. Blitz has no Short/Medium/Long length tiers (its budget is fixed, not selectable) but does use the same 1-6 Performance Rating scale as MA/Sugarcoat, added back after an initial version shipped without any rating at all.
**Reason:** Requested directly, from a provided spec (purpose, immutable rules, and 10 example writing models). The shuffle bag, the 70-120 (down from an initial 80-120) word floor, and the anti-copying instruction/validator were added after review identified: plain randomness could produce visible repeats; some lessons genuinely don't have 80+ words of honest material; and the model risked reproducing a selected example's sentences too closely over many generations rather than genuinely rewriting from the transcript. Rating was removed from Blitz's first version on the (reasonable, but ultimately wrong) assumption that a "quick neutral comment" protocol didn't need one — restored once actual use showed it should carry the same rating as the other protocols for workflow consistency.
**Constitution article:** Protocol Evolution Rule (shipped and explicitly locked as V1.0, not silently iterated on afterward); Facts vs. Judgment (the shuffle bag's `forcedModelKey` test-only override is a Facts-layer testing hook, never used by real generations, matching the same pattern already established for Short/Medium/Long's backward-compatible default); Bug Rule (a real usage-stats crash was found and fixed the same day Blitz shipped — `recordGeneration()` didn't know about the new protocol key — see the following entry).

### 2026-07-06 (same day, follow-up)
**Decision:** Fixed a crash in the cumulative usage-stats tracker: `loadUsageStats()`'s defaults object and its localStorage-merge logic only knew about `MA`, `OF`, and `MS` — never updated when Blitz was added — so `recordGeneration('BLITZ', cost)` threw trying to read `.reports` off of `undefined`, surfacing to the teacher as "Could not generate feedback. Cannot read properties of undefined (reading 'reports')" after a real Blitz generation had already succeeded. Added `BLITZ` to the defaults/merge logic and to the `kidbusterStats()` console summary. Added a permanent regression test (`tests/test-usage-stats.cjs`) written generically over `Object.keys(PROTOCOLS)` rather than hardcoding the four current protocol names, specifically so this class of bug — a protocol registered in `PROTOCOLS` but missed in some other hand-maintained enumeration elsewhere in the file — can't silently reappear for a future fifth protocol either.
**Reason:** Confirmed with certainty from a real generation (teacher "Manny", student "Alex") that produced this exact error message after Blitz otherwise worked correctly.
**Constitution article:** Bug Rule (confirmed bug with a certain root cause, fixed on sight, no threshold needed); Article Zero (a report that generates successfully but then fails to save its own usage stats is a broken promise about "trusted" tooling, even though the report itself was fine).

### 2026-07-06
**Decision:** Strengthened how "Special remarks" (the UI's label; the prompt text for MA/Sugarcoat/OF calls the same input "Teacher Notes") are handled across all four protocols. Previously, MA/OF had one generic line ("Always incorporate any additional Teacher Notes... Teacher Notes always override transcript interpretation") with no guidance on priority or placement; Sugarcoat had an equivalent one-line rule in its own voice; Blitz had no handling at all. Rewrote all four to explicitly state that remarks describing a behavioral or disciplinary concern are high-priority factual input, not optional color, and must be clearly reflected — specifically within the Parent Note section for MA/Sugarcoat (ratings 1-2.5), within Areas for Improvement for OF (which has no Parent Note concept), or as the "one area needing practice" line for Blitz. Also fixed a silent terminology mismatch: the outgoing user message is now built to explicitly say `Special remarks (these are the "Teacher Notes" referenced in your instructions) to incorporate:` for MA/Sugarcoat/OF, rather than relying on the model to infer the two field names refer to the same input.
**Reason:** Reported directly — a teacher-supplied remark about a student's behavior (tied to a low rating) hadn't visibly come through in a prior generation. Could not build a validator check that verifies arbitrary free-text remarks were actually reflected in the output (that requires understanding open-ended prose, not something pattern-matching can judge), so this is a prompt-engineering fix only; flagged as needing a live test with a real behavioral remark to confirm in practice, the same way the length-tier trim rule needed a real Short-tier generation to confirm.
**Constitution article:** Article Zero (a teacher-supplied fact about what actually happened in the lesson silently failing to surface is a direct failure of "trusted judgment," especially for the cases — low ratings, behavioral concerns — where a parent-facing note matters most); deliberate restraint against speculative validator engineering (no attempt to mechanically verify open-ended prose content, consistent with prior decisions to defer semantic checks until they're actually buildable).

### 2026-07-06
**Decision:** Added a permanent automated test suite (`tests/`, 8 files, ~230 checks) and CI (`.github/workflows/test.yml`, running on every push/PR against Node 18.x and 20.x). Tests extract `KidbusterCore` (and the UI-layer usage-stats block) directly out of the live `index.html` at run time — via locating the exact IIFE boundaries and evaluating it in an isolated scope — rather than maintaining a hand-copied snapshot that could silently drift out of sync with the real app.
**Reason:** Every feature shipped earlier in this log (length tiers, the trim rule, the wolf-emoji condition, Blitz, Special Remarks) had been verified ad hoc during development but the verification itself lived nowhere permanent — it would have had to be redone from scratch to catch a future regression in any of it. Coverage was written for both newly-added logic and previously-untested foundational behavior in MA/Classic and OF (star count, Parent Note gating, section order/limits, sign-offs) specifically because those are the oldest, most field-tested parts of the app and had zero direct test coverage despite that.
**Constitution article:** Article Zero (a validator the project has trusted for months but never actually tested itself is a gap in "trusted judgment," not just an engineering nicety); Bug Rule (the usage-stats test exists specifically because this exact class of bug already happened once — see the Blitz usage-stats entry above — and is written generically so it keeps catching that class of bug for any protocol added in the future, not just the ones that exist today).

### 2026-07-06
**Decision:** Added a Student pronoun toggle (Boy/Girl) to OF only — the one protocol written entirely in the third person about the student, and therefore the only one that actually needs a pronoun. ### 2026-07-06

**Decision:** Shifted from adding new architecture to validating and hardening the existing monetization flow. Two real reliability gaps were found and fixed, not just tested:
1. **Webhook replay/duplicate protection.** Lemon Squeezy's signature scheme (HMAC over the raw body, no timestamp/nonce) proves a webhook was genuinely signed, but not *when* — a validly-signed request can be resent later, either legitimately (Lemon Squeezy itself retries up to 3 times on a non-200) or by replaying a captured request. Added a SHA-256 hash of every processed webhook body to Vercel KV (30-day TTL) — a byte-identical body arriving again is recognized and skipped, handling duplicate delivery and replay identically without needing to tell them apart. The dedup *check* fails open (proceeds with processing) if KV itself can't be reached, deliberately prioritizing never dropping a real payment over avoiding a rare double-application — the license operations themselves are idempotent upserts, so a double-application from a dedup-check outage is a much smaller risk than silently losing a real payment.
2. **Write-order safety in `activateOrRenewPro`.** KV offers no multi-key transactions, so if a multi-write operation fails partway through, which write happened *first* determines which inconsistent state is possible — not whether one is possible at all. Reordered so the payment-customer/email indexes are written first and the actual license record (the thing that sets `plan: 'pro'`) is written last: if the last write never happens, the customer simply isn't upgraded yet (safe, retryable — a webhook retry naturally fixes it). The reverse order was the real risk: a license already marked Pro with a missing customer-id index would be permanently undiscoverable by any future cancellation webhook, leaving that customer stuck on Pro forever with no automatic way to downgrade them. `getOrCreateFreeLicense` deliberately keeps the opposite order (license record first, then index) — reasoned through separately, since a fresh signup's failure modes are different: worst case there, a missing index just means a future duplicate signup, not a permanently-stuck paid account.

Also raised `max_tokens` from 1500 to 3000 — measured Beida's realistic worst-case combined output (Learning Content + Performance near their platform-stated maximums) at ~1526 tokens, already over the old limit with zero margin.
2. **Sissy → 🌻, extended to every protocol that already had this mechanism** (Classic, Sugarcoat, OF, and now Beida), following the exact same `defaultTeacherSignatureEmoji()` pattern Codex built for Faye. Beida required a small addition specifically: since its greeting is generated prose rather than a literal "Teacher {name}" line the other protocols can substitute into, the hidden emoji is now threaded through as an explicit instruction in the Teacher Fact line (`buildUserMessage`), with the validator's greeting regex updated to expect it. Blitz was deliberately left alone — its own validator hard-forbids emoji anywhere in its output, and adding a first-ever teacher signature just to hold one would mean walking back a rule that was locked on purpose, not extending an existing convention.
3. **Blitz Voice system.** `BLITZ_MODEL_POOL` restructured from a flat 10-entry array to a two-dimensional `{ focus: { label, warm, simple, polished } }` object — 10 Focus archetypes (unchanged identities, still drawn automatically from the same shuffle bag) crossed with 3 selectable Voices (Warm/default, Simple, Polished), chosen via a new UI dropdown. Generation pipeline, placeholders, and validation are untouched by design; the anti-copy shingle check now scans all 30 texts instead of 10, since the model was only shown one but nothing stops it from drifting toward a different one's wording. The full Warm library was rewritten from scratch first — the original 10 examples were all under Blitz's own stated 70-120 word target (44-57 words each) and only 3 of 10 ever demonstrated the "one area needing practice" pattern the prompt's own rules require when evidence supports it — then Simple and Polished were derived from Warm to keep a consistent identity across all three voices, per explicit instruction. Also corrected an unflagged drift partway through drafting: earlier rewrites had shifted to third-person ("[Student] had..."), but the original locked examples — and the requested recurring phrase "continued building your confidence" — only make sense addressed directly to the student in second person, so the final Warm/Simple/Polished library restored that voice.
**Reason:** All three were requested directly in sequence. The Voice system in particular followed an iterative content-drafting process across several turns — first a straight reinforcement pass (fixing length/balance issues in the existing 10), then a full tone rewrite per a detailed brief (warm, natural, parent-friendly, avoiding "corporate" phrasing), then the two-voice-per-focus architecture request — each verified programmatically (word counts, banned-phrase scans) before being shown, not just written and assumed correct.
**Constitution article:** Bug Rule (the original Blitz examples silently violated Blitz's own stated word-count rule — found by actually counting, not by inspection); Facts vs. Judgment (every rewritten example was checked against concrete, stated criteria — word count range, an explicit banned-phrase list — before being presented, rather than judged only by feel); Article Zero (naming the third-person drift as a real, unprompted mistake rather than quietly fixing it and moving on).
